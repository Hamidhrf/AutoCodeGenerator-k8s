# AutoCodeGenerator — Complete Kubernetes Deployment Guide

> **Repo:** https://github.com/BhanuPDas/AutoCodeGenerator  
> **VM:** Ubuntu 24.04, NVIDIA H100 NVL (95GB VRAM), Kubernetes v1.32, CRI-O runtime

---

## What Went Wrong (And Why)

This guide documents the full journey of deploying AutoCodeGenerator on Kubernetes, including all the problems encountered and how to avoid them next time.

### Problem 1: Source Code Had Hardcoded IPs

The original source code had hardcoded VM IPs that break in Kubernetes (where services talk to each other via DNS names, not IPs).

**Files fixed:**

| File | Old Value | New Value |
|---|---|---|
| `UI/code-generator/src/Content.tsx` | `http://172.22.174.173:4010/api/process-request` | `/api/process-request` |
| `UI/code-generator/src/App.tsx` | `http://172.22.174.173:3000` | `http://localhost:3000` |
| `code-generator/src/main/resources/application.properties` | All hardcoded IPs | K8s service DNS names |

**application.properties correct values:**
```properties
inference.url=http://inference-service:5050/generate
redis.url=redis://redis-service:6379
spring.datasource.url=jdbc:postgresql://postgres-service:5432/postgres
```

**nginx.conf** needed a proxy pass for `/api/` to forward requests to the Java backend:
```nginx
location /api/ {
    proxy_pass http://code-generator-service:4010;
}
```

### Problem 2: NVIDIA Driver Package Conflict

Running `sudo apt-get install -y nvidia-utils-590` pulled in packages from the Ubuntu repo that conflicted with the CUDA repo packages. This created a broken state where:
- `nvidia-driver-535` and `nvidia-driver-590` were both partially installed
- `nvidia-smi` stopped working after reboot
- `libnvidia-ml.so` version mismatched with the kernel driver

**Root cause:** Never mix Ubuntu repo nvidia packages (`nvidia-utils-590` from `ubuntu/main`) with CUDA repo packages (`nvidia-driver-590` from `developer.download.nvidia.com`).

### Problem 3: Snap Docker Cannot Access Host GPU Libraries

Docker installed via `snap` runs in a sandboxed environment and cannot access host libraries at `/usr/lib/x86_64-linux-gnu/`. This causes:
```
nvidia-container-cli: initialization error: load library failed: libnvidia-ml.so.1: no such file
```

**Fix:** Remove snap Docker and install system Docker via `get.docker.com`.

### Problem 4: CDI Spec Referenced Missing MPS Binaries (Docker)



The CDI spec generated by `nvidia-ctk` included mount entries for `nvidia-cuda-mps-control` and `nvidia-cuda-mps-server`. When these binaries weren't present, Docker containers failed to start with:
```
failed to fulfil mount request: open /usr/bin/nvidia-cuda-mps-control: no such file or directory
```

### Problem 5: CRI-O nvidia.toml Placed in Wrong Directory

`nvidia-ctk runtime configure --runtime=crio` writes to `/etc/crio/conf.d/99-nvidia.toml` but CRI-O only reads drop-in configs from `/etc/crio/crio.conf.d/`. This means the nvidia runtime was never loaded by CRI-O, causing:
```
failed to find runtime handler nvidia from runtime list map[crun:... runc:...]
```

**Fix:** Copy the config to the correct directory:
```bash
sudo cp /etc/crio/conf.d/99-nvidia.toml /etc/crio/crio.conf.d/99-nvidia.toml
```

### Problem 6: CRI-O nvidia.toml Missing monitor_path

The config generated by `nvidia-ctk` was missing the `monitor_path` field required by CRI-O, causing it to fail to start:
```
failed to translate monitor fields for runtime nvidia: exec: "conmon": executable file not found in $PATH
```

**Fix:** Manually write the correct config with monitor_path:
```bash
sudo tee /etc/crio/crio.conf.d/99-nvidia.toml <<EOF
[crio.runtime.runtimes.nvidia]
runtime_path = "/usr/bin/nvidia-container-runtime"
runtime_type = "oci"
monitor_path = "/usr/libexec/crio/conmon"
EOF
```

### Problem 7: NVIDIA Device Plugin Could Not Find nvidia RuntimeClass

The device plugin pod itself ran with the default `runc` runtime and couldn't initialize NVML. The fix required:
1. Creating a `nvidia` RuntimeClass in Kubernetes
2. Adding `runtimeClassName: nvidia` to the inference deployment
3. Using a privileged device plugin daemonset that mounts the host `/` as `/driver-root`

**RuntimeClass:**
```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: nvidia
handler: nvidia
```

### Problem 8: NVIDIA Device Plugin Image Too Old for Host glibc

The standard `v0.17.0` device plugin image had a glibc version incompatible with Ubuntu 24.04, causing:
```
nvidia-device-plugin: symbol lookup error: libc.so.6: undefined symbol: __tunable_is_initialized
```

**Fix:** Use `v0.17.3` with `privileged: true` and mount host `/` as `/driver-root`.

### Problem 9: HuggingFace Token Lacked Model Access

Even with a valid HF token, the inference pod crashed because the token's account hadn't been granted access to the gated `meta-llama/CodeLlama-13b-Instruct-hf` model:
```
OSError: You are trying to access a gated repo... Access to model is restricted
```

**Fix:** Visit https://huggingface.co/meta-llama/CodeLlama-13b-Instruct-hf, request access, wait for approval, then update the K8s secret with a fresh token.

### Problem 10: Hardcoded Redis IP in Inference Code

`inference_13b_batch_c5.py` and `inference_13b_gptq_c4.py` had a hardcoded Docker bridge IP for Redis:
```python
rd = redis.Redis(host="172.17.0.1", port=6379, decode_responses=True)
```

**Fix:** Replace with the K8s service name:
```bash
sed -i 's/redis.Redis(host="172.17.0.1"/redis.Redis(host="redis-service"/g' inference_13b_batch_c5.py
sed -i "s/redis.Redis(host='172.17.0.1'/redis.Redis(host='redis-service'/g" inference_13b_gptq_c4.py
```

Then rebuild and push the inference image.

## Prerequisites Checklist

Before starting, verify these are all true:

```bash
# 1. nvidia-smi works
nvidia-smi

# 2. Docker (system, not snap) is installed
docker --version
which docker   # Should be /usr/bin/docker, NOT /snap/bin/docker

# 3. GPU works inside Docker
sudo docker run --rm --gpus all nvidia/cuda:12.6.0-base-ubuntu24.04 nvidia-smi

# 4. Kubernetes is running
kubectl get nodes   # Should show Ready

# 5. nvidia.com/gpu is allocatable in K8s
kubectl describe node | grep -A5 Allocatable | grep nvidia
```

All five must pass before deploying.

## Part 1 — Fix NVIDIA Stack (If Starting Fresh)

### Step 1.1 — Clean Install NVIDIA Driver

```bash
# Remove any conflicting packages first
sudo dpkg --remove --force-all nvidia-driver-535 nvidia-driver-590 \
  libnvidia-compute-535 libnvidia-compute-590 \
  xserver-xorg-video-nvidia-535 nvidia-compute-utils-535 \
  nvidia-utils-535 libnvidia-decode-535 2>/dev/null

sudo apt-get update
sudo apt-get install -y nvidia-driver-580   # Use whatever version is current
sudo reboot

# After reboot, verify
nvidia-smi
```

### Step 1.2 — Install NVIDIA Container Toolkit

```bash
sudo apt-get install -y nvidia-container-toolkit nvidia-utils-580

# Verify
nvidia-ctk --version
ls /usr/bin/nvidia-cuda-mps-control   # Must exist before generating CDI spec
```

### Step 1.3 — Generate CDI Spec

```bash
sudo mkdir -p /var/run/cdi /etc/cdi

# Generate spec (only after MPS binaries exist)
sudo nvidia-ctk cdi generate --output=/etc/cdi/nvidia.yaml --mode=nvml
sudo cp /etc/cdi/nvidia.yaml /var/run/cdi/nvidia.yaml

# Verify no missing MPS entries
grep -n "mps" /etc/cdi/nvidia.yaml   # Should show lines with the binaries
```

### Step 1.4 — Configure CRI-O

```bash
# IMPORTANT: nvidia-ctk writes to wrong directory — write config manually instead
sudo tee /etc/crio/crio.conf.d/99-nvidia.toml <<EOF
[crio.runtime.runtimes.nvidia]
runtime_path = "/usr/bin/nvidia-container-runtime"
runtime_type = "oci"
monitor_path = "/usr/libexec/crio/conmon"
EOF

sudo systemctl restart crio
sudo systemctl status crio | head -5   # Must show active (running)

# Fix ldconfig — must have @ prefix for CRI-O
grep "ldconfig" /etc/nvidia-container-runtime/config.toml
# Should show: ldconfig = "@/sbin/ldconfig.real"
# If missing @, add it:
sudo sed -i 's|ldconfig = "/sbin/ldconfig.real"|ldconfig = "@/sbin/ldconfig.real"|' \
  /etc/nvidia-container-runtime/config.toml
```

### Step 1.5 — Install System Docker (NOT Snap)

```bash
# Remove snap docker if present
sudo snap remove docker --purge

# Install system docker
curl -fsSL https://get.docker.com | sudo sh
sudo systemctl enable docker
sudo systemctl start docker

# Configure nvidia runtime for docker
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker

# Test GPU in docker
sudo docker run --rm --gpus all nvidia/cuda:12.6.0-base-ubuntu24.04 nvidia-smi
```

### Step 1.6 — Enable GPU in Kubernetes

```bash
# Create nvidia RuntimeClass
cat <<EOF | kubectl apply -f -
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: nvidia
handler: nvidia
EOF

# Deploy device plugin with privileged access and host driver root mount
cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: nvidia-device-plugin-daemonset
  namespace: kube-system
spec:
  selector:
    matchLabels:
      name: nvidia-device-plugin-ds
  template:
    metadata:
      labels:
        name: nvidia-device-plugin-ds
    spec:
      tolerations:
      - key: nvidia.com/gpu
        operator: Exists
        effect: NoSchedule
      priorityClassName: system-node-critical
      hostPID: true
      containers:
      - image: nvcr.io/nvidia/k8s-device-plugin:v0.17.3
        name: nvidia-device-plugin-ctr
        env:
        - name: DEVICE_DISCOVERY_STRATEGY
          value: "nvml"
        - name: NVIDIA_DRIVER_ROOT
          value: "/driver-root"
        securityContext:
          privileged: true
        volumeMounts:
        - name: device-plugin
          mountPath: /var/lib/kubelet/device-plugins
        - name: dev
          mountPath: /dev
        - name: driver-root
          mountPath: /driver-root
      volumes:
      - name: device-plugin
        hostPath:
          path: /var/lib/kubelet/device-plugins
      - name: dev
        hostPath:
          path: /dev
      - name: driver-root
        hostPath:
          path: /
EOF

# Wait and verify GPU is allocatable
sleep 20
kubectl describe node ubuntu | grep -A12 Allocatable | grep nvidia
# Should show: nvidia.com/gpu: 1
```

## Part 2 — Build and Push Images

```bash
cd ~/AutoCodeGenerator

# Make sure you're logged into Docker Hub first
docker login -u hamidhrf

# Inference (requires HuggingFace token with access to CodeLlama-13B)
cd inference_pipeline
sudo docker build --build-arg HF_TOKEN=<your-hf-token> \
  -t hamidhrf/autocodegens-inference:latest .
sudo docker push hamidhrf/autocodegens-inference:latest

# Java service
cd ../code-generator
sudo docker build -t hamidhrf/autocodegens-java:latest .
sudo docker push hamidhrf/autocodegens-java:latest

# UI (nginx with /api/ proxy to java backend)
cd ../UI/code-generator
sudo docker build -t hamidhrf/autocodegens-ui:latest .
sudo docker push hamidhrf/autocodegens-ui:latest
```

> **Security:** Regenerate your HuggingFace token after each build. Never commit it to git.

## Part 3 — Deploy to Kubernetes

### Step 3.1 — Namespace, Secret, and RuntimeClass

```bash
kubectl apply -f k8s/namespace.yaml

kubectl create secret generic hf-secret \
  --from-literal=token=<your-hf-token> \
  -n autocodegens

# Verify RuntimeClass exists (created in Step 1.6)
kubectl get runtimeclass nvidia
```

> **HuggingFace access:** Before deploying, make sure your HF account has been granted access to `meta-llama/CodeLlama-13b-Instruct-hf` at https://huggingface.co/meta-llama/CodeLlama-13b-Instruct-hf. Approval is usually instant but required.

### Step 3.2 — Deploy Dependencies

```bash
# Postgres (uses hostPath for persistence)
kubectl apply -f k8s/postgres/deployment.yaml
kubectl apply -f k8s/postgres/service.yaml

# Redis
kubectl apply -f k8s/redis/deployment.yaml
kubectl apply -f k8s/redis/service.yaml

# Wait for both to be Running
kubectl get pods -n autocodegens -w
```

### Step 3.3 — Deploy Application Services

```bash
kubectl apply -f k8s/inference/deployment.yaml
kubectl apply -f k8s/inference/service.yaml
kubectl apply -f k8s/java-service/deployment.yaml
kubectl apply -f k8s/java-service/service.yaml
kubectl apply -f k8s/ui/deployment.yaml
kubectl apply -f k8s/ui/service.yaml
```

### Step 3.4 — Verify

```bash
kubectl get pods -n autocodegens
# Expected:
# inference      1/1  Running  (takes 5-10 min to load model)
# code-generator 1/1  Running
# ui             1/1  Running
# redis          1/1  Running
# postgres       1/1  Running

kubectl get svc -n autocodegens
```

### Step 3.5 — Test

```bash
# Test inference directly
kubectl port-forward svc/inference-service 5050:5050 -n autocodegens &
curl -X POST http://localhost:5050/generate \
  -H 'Content-Type: application/json' \
  -d '{"prompt": "Write a hello world in Python"}'

# Access UI in browser
echo "http://$(hostname -I | awk '{print $1}'):30090"
```

## Troubleshooting Quick Reference

### Inference pod stuck in Pending
```bash
kubectl describe pod -n autocodegens -l app=inference | grep -A10 Events
# If "Insufficient nvidia.com/gpu" → GPU not available in K8s
# Fix: Re-run Part 1 steps 1.3–1.6
```

### nvidia-smi fails after reboot
```bash
# Driver conflict — clean reinstall
sudo dpkg --remove --force-all nvidia-driver-535 nvidia-driver-590
sudo apt-get install -y nvidia-driver-580
sudo reboot
```

### Docker GPU fails with "libnvidia-ml.so not found"
```bash
# Check if using snap docker (bad) or system docker (good)
which docker   # Must NOT be /snap/bin/docker
# If snap: sudo snap remove docker --purge && curl -fsSL https://get.docker.com | sudo sh
```

### CDI spec MPS mount error
```bash
# Ensure MPS binaries exist BEFORE generating CDI spec
ls /usr/bin/nvidia-cuda-mps-control   # Must exist
sudo apt-get install -y nvidia-utils-580   # Installs MPS binaries
# Then regenerate: sudo nvidia-ctk cdi generate --output=/etc/cdi/nvidia.yaml --mode=nvml
```

### Inference pod crashes (OOMKilled or model load failure)
```bash
kubectl logs -n autocodegens -l app=inference --previous
# CodeLlama-13B needs ~27GB VRAM minimum
# H100 NVL has 95GB — should be fine
# Check HF token is valid and has model access
```

### Inference pod crashes with "Access to gated repo restricted"
```bash
# 1. Visit https://huggingface.co/meta-llama/CodeLlama-13b-Instruct-hf and request access
# 2. Generate a fresh token at https://huggingface.co/settings/tokens
# 3. Update the K8s secret
kubectl delete secret hf-secret -n autocodegens
kubectl create secret generic hf-secret --from-literal=token=<new-token> -n autocodegens
kubectl rollout restart deployment/inference -n autocodegens
```

### Java service can't reach inference/redis/postgres
```bash
# Verify service DNS names match application.properties
kubectl get svc -n autocodegens
# Names must match: inference-service, redis-service, postgres-service
```

## Notes

- The inference pod takes **5–10 minutes** to start on first run (downloading and loading CodeLlama-13B)
- Postgres data persists at `/home/user/autocodegens/postgres-data` on the host across pod restarts
- Never mix Ubuntu repo nvidia packages with CUDA repo nvidia packages
- Always use system Docker (`/usr/bin/docker`), never snap Docker, for GPU workloads on this VM
- HuggingFace token must have accepted the CodeLlama-13B model license at huggingface.co
- The inference deployment **must** have `runtimeClassName: nvidia` in its pod spec — without it the container runs with runc and can't access the GPU
- CRI-O drop-in configs must go in `/etc/crio/crio.conf.d/` — the `/etc/crio/conf.d/` directory is NOT read by CRI-O
- The nvidia device plugin must use image `v0.17.3` or newer on Ubuntu 24.04 (older images have glibc incompatibility)
